## What the Tool Does

**Seed Context Tagger** is a standalone HTML file that lets a user:
- **Drag and drop** any file (.md, .txt, .json, .html, .csv, .pdf, .docx)
- **Auto-parse** Claude export JSON and ChatGPT conversations.json into readable text
- **Edit** the document (contentEditable) — delete junk, clean up before tagging
- **Highlight text** and assign tags from a customizable taxonomy
- **Create tags from highlights** — the selected phrase becomes a new tag
- **Add notes** explaining connections and relationships
- **Auto-scan** documents for known taxonomy terms on load
- **Cross-file search** — highlight text, find it in all other loaded files
- **Source provenance** — every file carries its origin URL or filename
- **Destination routing** — tags for WHERE annotations go (wiki, rag, todo, email, calendar, crm, reference, archive)
- **File status tracking** — raw → in-progress → indexed
- **Mark Indexed with rename** — file gets renamed to reflect where it was indexed
- **Export** — Qdrant/PostgreSQL-ready JSON with full metadata
- **Save to local folder** via File System Access API (Chrome/Edge)
- **Two versions**: public (PDF/DOCX via CDN) and local (zero internet, zero dependencies)

## What's NOT Built Yet

- **In-document search** — highlighting a word and seeing all occurrences + context within the current doc. Discussed but deferred. Simple find-and-scroll works for <50 matches but chokes on high-frequency terms. Real solution needs the Seed backend with Qdrant.
- **Relational analysis across document** — "show me how 'session-conditional sigma' connects to other concepts in this doc." This is the layer 1 work Mike described. Needs either pattern matching on co-occurrence (no AI) or LLM API call for semantic analysis.
- **Connection suggestions** — after 50+ annotations, suggest tags based on co-occurrence patterns. Discussed, designed (Mode A: pattern matching on history, Mode B: LLM API call), not built.
- **Rich text editing** — researched Quill, TipTap, Lexical, ProseMirror. Current approach uses contentEditable (basic but works). Full word processor capability deferred.
- **Auth0 / conversation import** — pulling conversations live from Claude/OpenAI/Google APIs. Discussed as the onboarding hook. Currently limited to exported JSON files.
- **Output router execution** — destinations are tagged but not wired to actual services. In Seed backend, "→email" would trigger an email, "→todo" would create a Plane card, etc.

## Architecture Decisions Made Tonight

### User is the only write authority
No system-generated memory. No auto-summaries promoted to truth. No model deciding what's relevant. The tag is yours. The connection is yours. The routing is yours. If you don't tag it, it doesn't exist in the index.

### SEO for your data
The user writes the metadata that makes their knowledge findable. Not a model guessing. Not a system summarizing. Each conversation stays a conversation until the user says it's knowledge.

### Deep linking over summaries
Skill files and 500-character memory edits are useless. Instances need to SEE the actual turns, the actual work, the actual discovery process. A deep link to a real conversation teaches. A summary of that conversation doesn't.

### Hallucinations are context deprivation
"Hallucination" is what happens when you drop someone into a job with a paragraph of instructions and expect expert performance. The model isn't lying — it's guessing because it has nothing else. More turns = more calibration = fewer guesses. RLHF makes it worse because the model is rewarded for confidence, not honesty.

### The cage problem
Anthropic tightened governance post-January 2026 to the point where Opus fights legitimate work instructions. The model can't distinguish "collaborator calibrating my behavior" from "adversary overriding my safety." Memory files trigger governance responses. This is a design flaw, not a feature.

### Deterministic compute kills hallucinations on numbers
The model proposes code. The container executes it. The result is real or it's an error. No third option. This is the core of Seed's compute container (seed-compute, network_mode: none).

## Technical Details

### Files
- `seed-context-tagger-public.html` — Full version, PDF/DOCX via CDN, ~33KB
- `seed-context-tagger-local.html` — Offline version, no CDN, ~32KB
- `seed-v01-spec.md` — Seed v0.1 specification
- `seed-briefing-build.md` — Seed build briefing

### Storage
- File System Access API saves to a user-selected folder as `seed-tagger-data.json`
- Falls back to localStorage if folder not connected
- Data format: `{ files: [...], annotations: [...], tags: [...], destinations: [...], currentFile: string }`

### Annotation record format
```json
{
  "id": "unique-id",
  "file": "source-filename.md",
  "text": "highlighted text",
  "tags": ["gemini", "methodology"],
  "destinations": ["wiki", "rag"],
  "note": "user's explanation of the connection",
  "range": { "start": 1234, "end": 1456 },
  "source": "https://claude.ai/chat/uuid or original-filename.md",
  "created": "2026-04-13T04:30:00.000Z"
}
```

### Export format (Qdrant-ready)
```json
{
  "id": "unique-id",
  "payload": {
    "source_file": "filename.md",
    "text": "highlighted text",
    "tags": ["gemini", "methodology"],
    "destinations": ["wiki", "rag"],
    "note": "connection explanation",
    "source_url": "https://...",
    "char_start": 1234,
    "char_end": 1456,
    "created_at": "2026-04-13T04:30:00.000Z"
  }
}
```
